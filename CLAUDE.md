## üèóÔ∏è Project Architecture

### Core Principles
1. **No LLM/AI calls** - Deterministic, heuristic-based analysis only
2. **No network calls** - Fully offline, local-first
3. **Evidence-based** - Never invent "why", only report what changed
4. **Extensible** - Profile-based analyzer architecture

### Data Flow
```
CLI ‚Üí Git Collector ‚Üí ChangeSet ‚Üí Profile ‚Üí Analyzers ‚Üí Findings ‚Üí Renderer ‚Üí Output
```

### Available Commands
The CLI has multiple commands for different use cases:
- **`integrate`** - Generate full PR description with findings (default command)
- **`pr-body`** - Alias for integrate (backward compatibility)
- **`pretty`** - Pretty terminal output for quick review
- **`facts`** - JSON output with prioritized highlights (v2.0+)
- **`risk-report`** - Risk analysis with scoring (0-100) and flags
- **`dump-diff`** - Clean git diff output optimized for tokens
- **`zoom`** - Verify specific findings or changes
- **`snap`** - Snapshot management for delta mode

### Highlights System (v2.0+)
The **highlights** feature generates prioritized summary bullets from findings. It's used in the `facts` command to provide a concise, impact-ordered summary.

**Priority ordering** (highest to lowest):
1. **Blast radius** - High/medium impact analysis
2. **Breaking changes** - Config/API surface changes
3. **Risk/security warnings** - High-risk flags, destructive SQL, DB migrations, CI security
4. **Lockfile mismatches** - Package lock inconsistencies
5. **General changes** - Infrastructure, dependencies, routes, API contracts
6. **API surface changes** - Stencil, monorepo, environment variables
7. **Test coverage** - Test changes and gaps
8. **Informational** - General findings

Highlights are generated by `src/commands/facts/highlights.ts` and provide a quick overview for AI agents and developers.

### Available Profiles
Branch-narrator supports multiple framework profiles with specialized analyzers:
- **`default`** - General-purpose profile with all analyzers
- **`sveltekit`** - SvelteKit-specific detection (routes, layouts, endpoints)
- **`next`** - Next.js App Router and Pages Router detection
- **`react`** - React Router detection (JSX routes and data routers)
- **`vue`** - Vue Router detection (file-based and configured routes)
- **`astro`** - Astro pages and API routes detection
- **`stencil`** - Stencil component API detection (props, methods, events, slots)
- **`library`** - Library-focused profile for packages

## üíª Coding Standards & Patterns

### TypeScript Rules

```typescript
// ‚úÖ ALWAYS use explicit types for parameters and returns
function analyze(changeSet: ChangeSet): Finding[] {
  return [];
}

// ‚ùå NEVER use implicit types
function analyze(changeSet) {
  return [];
}
```

**Type vs Interface:**
- Use `interface` for object shapes
- Use `type` for unions and primitives
- Use discriminated unions with `type` field as discriminator

**Imports:**
- Always use `.js` extensions for local imports (ESM requirement)
- Import order: Node built-ins ‚Üí External packages ‚Üí Internal modules

```typescript
// ‚úÖ Correct import with .js extension
import { foo } from "./bar.js";
import type { Analyzer, ChangeSet } from "../core/types.js";

// ‚ùå Wrong - missing .js extension
import { foo } from "./bar";
```

### Naming Conventions

| Element | Convention | Example |
|---------|------------|---------|
| Files | kebab-case | `route-detector.ts` |
| Types/Interfaces | PascalCase | `RouteChangeFinding` |
| Functions | camelCase | `detectMethods()` |
| Constants | UPPER_SNAKE | `CRITICAL_PACKAGES` |
| Analyzers | camelCase + Analyzer | `routeDetectorAnalyzer` |

### Analyzer Pattern (Template)

```typescript
import type { Analyzer, ChangeSet, Finding } from "../core/types.js";

export const myAnalyzer: Analyzer = {
  name: "my-analyzer",

  analyze(changeSet: ChangeSet): Finding[] {
    const findings: Finding[] = [];

    for (const file of changeSet.files) {
      // Detection logic here
      if (matchesPattern(file.path)) {
        findings.push({
          type: "my-finding",
          // ... finding properties
        });
      }
    }

    return findings;
  },
};
```

**Key principles:**
- Analyzers are **pure functions**: `(changeSet: ChangeSet) => Finding[]`
- Analyzers must be **deterministic** and **stateless**
- Return empty array if no findings (never null/undefined)
- Include evidence in findings (file paths, reasons, etc.)

### Finding Type Pattern

```typescript
// 1. Define interface in src/core/types.ts
export interface MyFinding {
  type: "my-finding";  // Discriminator
  files: string[];
  reason: string;
}

// 2. Add to Finding union in src/core/types.ts
export type Finding =
  | FileSummaryFinding
  | RouteChangeFinding
  // ... existing
  | MyFinding;
```

All errors extend `BranchNarratorError` with an `exitCode` property.

## üìã Complete Workflow Example: Adding a New Analyzer

Let's walk through a complete example of adding a Tailwind CSS analyzer:

### Step 1: Create the Analyzer
```typescript
// src/analyzers/tailwind.ts
import type { Analyzer, ChangeSet, Finding } from "../core/types.js";

export const tailwindAnalyzer: Analyzer = {
  name: "tailwind-analyzer",

  analyze(changeSet: ChangeSet): Finding[] {
    const findings: Finding[] = [];

    for (const file of changeSet.files) {
      if (file.path === "tailwind.config.js" || file.path === "tailwind.config.ts") {
        findings.push({
          type: "tailwind-config-change",
          file: file.path,
          status: file.status,
        });
      }
    }

    return findings;
  },
};
```

### Step 2: Add Finding Type
```typescript
// In src/core/types.ts - add interface
export interface TailwindConfigChangeFinding {
  type: "tailwind-config-change";
  file: string;
  status: FileStatus;
}

// In src/core/types.ts - add to union
export type Finding =
  | FileSummaryFinding
  | RouteChangeFinding
  // ... existing
  | TailwindConfigChangeFinding;
```

### Step 3: Export Analyzer

### Step 4: Add to Profile
